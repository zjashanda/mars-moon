## 2. 先给结论

### 2.1 当前 23 个 FAIL 不是 23 个独立问题

按“唯一问题簇”看，当前更接近下面 6 类：

1. 上电欢迎语未出现。
2. 默认交互态超时不是需求的 20s，而是约 15.69s。
3. 切换唤醒词后，交互态超时进一步缩短到约 13.69s~14.14s。
4. 三个设置模式的超时路径不符合当前规格口径，实测约 16.33s~16.41s，且唤醒词设置超时走到 `TONE_ID_11`。
5. 音量掉电后未恢复默认值，表现为掉电前调低 1 档后，掉电后仍保留该档位。
6. 遥控器配对入口协议/播报与词表、参考用例不一致，导致配对窗口相关验证整体失效。

### 2.2 当前最大的问题不是“跑不通”，而是“可追溯性不够精确”

当前链路已经能稳定完成 `prepare -> build -> burn -> run`，但是“需求 -> REQ_* -> case -> 结果”的映射还不够严谨，主要有 4 个结构性问题：

1. `REQ_*` 不是从原始需求逐条抽取出来的，而是 `tools/dooya_spec_builder.py` 中静态生成的模块模板。
2. 同一个模块下的所有 case 都挂了该模块下全部 `REQ_*`，导致一个 case fail 时，很难直接说是哪条需求 fail。
3. 一些 case 的前置状态没有显式恢复，PASS 不能完全证明它验证的是声明中的前置状态。
4. 一些负例用的是词表外无效词，PASS 只能证明“没有反应”，不能证明“设备识别到了但按照状态机正确拒绝”。

### 2.3 如果目标是“看需求表就知道验证状态”，现在还差 3 件事

1. 需求条目要有唯一源 ID，并标明来源是：主需求 / 补充说明 / 词表 / tone / 参考用例。
2. 每条需求要绑定“主验证用例”，不要再让一个 case 同时代表一个模块下所有需求。
3. 执行结果要回写成“需求视角”的状态，而不是只有 case 视角的状态。

## 3. 当前 traceability 的主要断点

### 3.1 `REQ_*` 是静态模板，不是从原始需求逐条生成

`tools/dooya_spec_builder.py:615` 的 `build_requirement_modules()` 直接固定生成 13 个模块与对应需求句子；`tools/dooya_spec_builder.py:752` 的 `build_coverage_matrix()` 再按模块顺序生成 `REQ_*`。

这带来两个直接后果：

- `BOOT`、`CTRL`、`PHRASE`、`SELECTOR` 这些模块会被直接放进覆盖矩阵，但它们并不都来自 `tmp/需求文档.md` 的显式条款。
- 覆盖矩阵看起来很完整，但它不是“原始需求逐条可追溯矩阵”，更像“当前工具理解下的一套通用模板矩阵”。

### 3.2 case 与 requirement 是“整模块绑定”，不是“逐条绑定”

`tools/dooya_case_builder.py` 中通过 `self.requirement_ids(module)` 把同模块全部 `REQ_*` 都挂到每一条 case 上。

结果就是：

- `TC_BASE_002`、`TC_BASE_003`、`TC_BASE_004` 都同时绑定 `REQ_BASE_001~003`。
- `TC_WAKEWORD_004~011` 都同时绑定 `REQ_WAKEWORD_001~004`。
- `TC_CTRL_001~007` 都同时绑定 `REQ_CTRL_001~005`。

这样虽然“形式上覆盖了”，但执行后无法直接回答“到底是哪条需求失败了”。

### 3.3 reference case 还是旧口径，存在历史残留冲突

`assets/reference/CSK5062_杜亚窗帘_测试用例v2.xlsx` 中基础唤醒超时仍是 15s 口径，而 `tmp/需求文档.md` 已经是 20s。说明参考用例资产本身是旧版本，不能继续当主权威来源。

这意味着后续同类产品必须明确来源优先级：

1. 主需求文档
2. 补充说明
3. 词表 / tone
4. 历史参考用例

只要旧参考用例与当前需求冲突，就只能作为参考，不能直接驱动 FAIL 判定。

3. 设备自检命令读取校验。

- `SELECTOR`

如果这些内容继续与主需求同级展示，会让人误以为“需求文档里本来就写了这些”，这会直接破坏 traceability。

## 5. 当前用例设计里最需要优先修正的点

### 5.1 一个 case 同时做“时间校验 + 状态回退校验”，归因会失真

典型例子：

- `TC_BASE_003`
- `TC_BASE_004`
- `TC_WORKMODE_003`
- `TC_CURTAINMODE_004`

这些 case 都先等 20s，再继续做后置动作。一旦“20s 不成立”，case 直接在 wait 动作失败，后面用于验证“是否退出设置态”“是否恢复默认行为”的动作根本没执行。

结果是：

- 我们知道“时间值不对”；
- 但并不知道“时间不对之外，状态回退是否也不对”。

所以同类产品后续必须拆成两条：

1. `timeout_value_case`：只量超时时间与超时播报；
2. `timeout_behavior_case`：等待“实际发生超时事件”后，再验证退出后的行为。

### 5.2 负例 case 不能一直用词表外的 `测试无效词`

`tools/dooya_case_builder.py:287` 现在统一把设置类负例词写成 `测试无效词`。从本轮实测看，相关 case 都是 `ASR=None`。

这会导致 PASS 含义变弱：

- 现在的 PASS 只能说明“设备没反应”；
- 但不能说明“设备在当前设置窗口下正确拒绝了一个已识别但不合法的输入”。

更稳的做法应该是：

- 唤醒词设置中，用一个“词表内、但不是候选唤醒词”的词作为负例；
- 工作模式设置中，用 `打开窗帘` 这类已识别但不属于模式候选的词作为负例；
- 窗帘模式设置中，用 `打开窗帘` 或 `打开布帘` 这类已识别但不属于模式选择词的词作为负例。

这样才能验证“状态机拒绝”而不是“ASR 没识别”。

### 5.3 `CTRL` 模块应该有 blocker 关系，不该把派生问题都记成独立 FAIL

本轮 `TC_CTRL_001` 已经证明配对入口不符合当前期望：

- 期望：`55 AA 05 01 12 00 80 AF`
- 实测：`55 AA 04 01 12 B2 17`
- 期望播报：`TONE_ID_6`
- 实测播报：`TONE_ID_0`

在这个前提下：

- `TC_CTRL_002`
- `TC_CTRL_003`
- `TC_CTRL_004`
- `TC_CTRL_005`
- `TC_CTRL_006`

都应该被视为“依赖未满足后的派生结果”，至少要能标成“BLOCKED by TC_CTRL_001”或“派生失败”，而不是继续作为 5 个同级独立 FAIL。

### 5.4 `UART` 基础能力验证不该依赖最不稳定的业务命令

`TC_UART_001`、`TC_UART_002` 当前都绑定 `配对遥控器`。这导致：

- 一旦配对协议与需求表不一致，UART 模块看起来就像失败了；
- 但实际上串口链路是通的，失败的是业务协议口径。

后续应至少拆成两层：

1. `UART_SMOKE`：用 `打开窗帘` / `清除遥控器` 这类稳定词验证“协议口有输出、日志字段完整”；
2. `UART_REMOTE_PAIR`：单独验证配对协议和被动播报链路。

### 5.5 默认状态类 case 有状态污染风险

当前最明显的是：

- `TC_CURTAINMODE_007` 先把设备切到布帘模式；
- 紧接着就执行 `TC_CURTAIN_001`；
- 但 `TC_CURTAIN_001` 的前置条件写的是“设备处于默认窗帘模式”，动作里却没有恢复默认模式。

这意味着 `TC_CURTAIN_001` 即便 PASS，也不能证明“默认窗帘模式成立”。

同类风险还包括：

- `TC_CURTAIN_001~003`
- `TC_PHRASE_001`
- 所有依赖“默认模式 / 默认音量 / 默认唤醒词”的 case

后续需要强制执行两种策略之一：

1. 每条 case 自己恢复 baseline；
2. 在模块切换点插入显式 baseline case，并让后续 case 依赖它。

当前我更建议第 1 种，因为更容易定位问题，也更利于跨产品复用。

## 6. 功能之间的联系，以及哪些问题会阻塞其他测试

### 6.1 交互态超时是跨模块基础能力

它不只是 `BASE` 的需求，还会影响：

- 唤醒词切换后的稳定性验证
- 三个设置模式的超时退出验证
- 配对窗口超时验证
- 连续唤醒、设置窗口等待、超时回退等很多用例的编排

所以“超时时间不对”属于跨模块基础缺陷，不是局部小问题。

### 6.2 默认唤醒词是全局恢复入口

默认唤醒词一旦失效，会直接阻塞：

- WAKEWORD 后续所有验证
- WORKMODE / CURTAINMODE / CURTAIN / CTRL 绝大多数语音入口
- 甚至 runner 自身的很多自动唤醒步骤

本轮默认唤醒词稳定可用，所以它不是当前阻塞点；但后续报告里应该把它单列为“全局前置能力”。

### 6.3 工厂恢复是关键 baseline 恢复手段

音量、工作模式、窗帘模式、唤醒词都依赖它做状态清理。凡是会修改持久化状态的用例，如果没有工厂恢复或等效 baseline，就容易污染后续结果。

### 6.4 遥控器配对是双通道联合功能

它同时依赖：

- 语音入口命令词
- 协议串口注入
- 日志口 receive msg 观测
- 被动播报

所以它天然就应该用“主入口 case + 依赖 case”的方式建模，而不适合和普通单命令词 case 同级平铺。

## 7. 当前 FAIL 里，哪些是阻塞项，哪些是独立 bug

### 7.1 会阻塞后续验证的项

#### 阻塞项 A：配对遥控器入口口径不一致

影响范围：

- `TC_UART_001`
- `TC_UART_002`
- `TC_CTRL_001~006`

这类问题更像“需求/词表/参考用例 与 现网协议不一致”的阻塞项。只要入口不统一，后续配对窗口、被动播报、窗口关闭都无法可靠验证。

#### 阻塞项 B：超时类 case 的第一断言失败

影响范围：

- `TC_BASE_003`
- `TC_BASE_004`
- `TC_WAKEWORD_004~010`
- `TC_WORKMODE_003`
- `TC_CURTAINMODE_004`
- 未来任何依赖固定 20s 的窗口类 case

这类不是“设备完全不可测”，而是“当前 case 的主前提先失败，导致后置断言失去意义”。

### 7.2 当前更像独立 bug 的项

- `TC_BOOT_001`：欢迎语缺失。
- `TC_POWER_003`：音量掉电保持行为与需求不符。
- `TC_BASE_002`：默认交互态超时值不符。
- `TC_WAKEWORD_011`：唤醒词设置超时值/超时路径与当前规格不符。

## 8. 如何判断是工具问题、固件问题、用例问题还是需求问题

### 8.1 需求/源文件问题

满足下面任一条，就优先归到“需求/源文件问题”：

- 主需求、补充说明、词表、tone、参考用例彼此冲突；
- 期望值只能从旧参考用例拿到，而主需求没有明确写；
- 需求原文本身有歧义或像笔误，例如窗帘模式第 5 条。

本轮典型：`配对遥控器` 协议口径。

### 8.2 用例问题

满足下面任一条，就优先归到“用例问题”：

- case 前置状态没有显式建立；
- 一个 case 同时验证两个独立断言，前一个失败导致后一个根本没跑；
- 负例词不在词表内，无法证明状态机拒绝；
- 基础能力验证绑在一个高风险业务命令上。

本轮典型：

- `TC_BASE_003` / `TC_BASE_004`
- `TC_WORKMODE_003`
- `TC_CURTAINMODE_004`
- `TC_UART_001` / `TC_UART_002`
- `TC_CURTAIN_001~003`

### 8.3 工具问题

满足下面任一条，才应优先怀疑工具：

- 串口读不到、协议口打不开、日志恢复失败；
- 音频播放失败；
- 同一 case 同口径多次复跑结果高度随机；
- 日志 / 协议截断严重，证据链本身不可信。

本轮没有看到这类主因。当前链路是稳定可复现的，所以 23 个 FAIL 里没有主因属于工具故障。

### 8.4 固件问题

满足下面任一条，可优先归到“固件问题”：

- 需求口径明确且源文件之间无冲突；
- 用例前置清晰、执行稳定可复现；
- 实测行为稳定偏离需求。

本轮更偏固件问题的典型有：

- 默认交互态超时不是 20s；
- 设置模式超时不符合当前规格口径；
- 掉电后音量未恢复默认；
- 欢迎语未播报（如果确认 tone/tone 表是本版本合同的一部分）。

## 9. 后续同类型产品应该怎么验证

### 9.1 先把需求拆成 4 类，而不是一股脑直接生成 case

1. `静态配置类`
   - 如 mic 增益、发音人、UART 编号、默认值。
   - 应优先用配置文件 / 构建产物校验。

2. `运行时状态机类`
   - 如唤醒超时、设置模式、模式切换、恢复出厂、掉电保持。
   - 应用实机用例验证。

3. `协议接口类`
   - 如 UART 发送、被动 receive msg、窗口时序。
   - 需要实机 + 协议口联合验证。

4. `回归语义类`
   - 如同义词、词条组、历史兼容词。
   - 这类是加分项或回归项，不应与主需求 acceptance 同层混算。

### 9.2 每条需求要有“主验证方案”与“辅助验证方案”

建议后续需求表最少增加这些列：

- 需求ID
- 来源（主需求 / 补充说明 / 词表 / tone / 参考用例）
- 需求内容
- 可观察类型（静态 / 运行时 / 协议 / 人工感知）
- 主验证方案
- 用例ID
- 最近结果（PASS / FAIL / BLOCKED / 未覆盖）
- 问题归因（需求 / 用例 / 工具 / 固件）
- 证据路径

### 9.3 每条 case 只承载一个核心断言

后续生成规则建议改成：

- 一个主需求条款 -> 一个主 case；
- 如果需要闭环，可带必要前置和后置动作；
- 但不要把“时间值”和“超时后行为”揉进同一条主 case。

### 9.4 有依赖关系的 case 要允许 BLOCKED，而不是一律 FAIL

特别是：

- 遥控器配对窗口类
- 需要先进入设置窗口的被动验证类
- 需要先通过基线 case 才能证明后续状态的 case

如果入口条件已经不成立，后续更适合标 `BLOCKED by XXX`。

### 9.5 每条依赖默认状态的 case 都要显式建基线

建议统一基线动作：

- 恢复出厂
- 等待稳定
- 默认唤醒一次
- 必要时再校验默认工作模式 / 默认窗帘模式 / 默认音量

这样后续看某条 case 失败时，才不会先怀疑是不是前一条 case 污染了状态。

## 10. 我认为最值得优先落地的优化顺序

### 第一优先级：先修 traceability，不先扩 case 数量

先解决这 4 件事：

1. 给原始需求逐条编号。
2. 给每条 case 标唯一主需求，而不是整模块绑定。
3. 把 derived module（PHRASE / SELECTOR / CTRL / BOOT）单独标来源。
4. 生成一份“需求视角结果表”，直接写明每条需求的当前状态。

### 第二优先级：修 case 设计缺陷

1. 拆分 timeout 时间值与 timeout 后行为。
2. 把设置类负例改成“词表内、状态机应拒绝”的词。
3. 给所有默认状态 case 显式补 baseline。
4. 给 `CTRL` / `UART` 建 blocker 关系。

### 第三优先级：补当前主需求的缺口

优先补：

1. mic 增益静态校验。
2. 发音人配置静态校验。
3. UART 内部编号配置静态校验。

这三个点不一定要靠 runner 才能做，但必须进入“需求可验证闭环”。

## 11. 这份分析对应到当前实机结果的最终判断

- 当前工具链可用，不是主因。
- 当前 23 个 FAIL 里，真正唯一问题簇大约 6 类，不应按 23 个独立问题理解。
- 当前最大的短板是 traceability 和 case 设计，不是执行能力。
- 如果目标是“根据需求能写出对应测试用例，并且执行后能直接判断需求是否正确”，那下一步最应该先做的是：
  - 重建需求源 ID；
  - 重建一对一主验证 case；
  - 把 BLOCKED / 派生失败 / 真 FAIL 分开；
  - 再把结果回填到需求表。
