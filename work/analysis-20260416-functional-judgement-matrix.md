# 功能判定矩阵（2026-04-16）

## 1. 总原则

- 先判功能是否成立，再判数值/协议/tone 是否与需求一致。
- 需求文档不是绝对真值；若功能成立但数值不一致，结论应是“功能通过 + 数值差异”，而不是直接阻断后续链路。
- `configSaved=save config success` 只作为辅助证据，不作为用例硬 FAIL 条件。
- 负例不能因为看到了 `keyword:` 或 `intentStr` 就判失败，必须看 DUT 是否真的执行了不该执行的动作。

## 2. 主功能的功能层判据

### 2.1 唤醒 / 基础会话
- 功能通过：`keyword:你好杜亚(或目标唤醒词)` + 唤醒 tone + `wake up ready to asr mode`，必要时再带 `MODE=1`。
- 功能失败：未形成完整唤醒链；仅出现 `keyword:`、仅出现 `wakeReady`、或仅出现孤立 tone 都不算完整唤醒。
- 超时退出通过：`TIME_OUT` + `MODE=0` + `evt msg -> exit`；tone 可为 `play id : 2`，也可能出现 `play id : 11`。
- 数值比对：唤醒窗口时长是否接近需求 20s。

### 2.2 恢复出厂
- 功能通过：`keyword:恢复出厂模式` 后出现 `restore factory mode`，并有确认 tone/protocol 任一可佐证；后置行为恢复默认。
- 强辅助日志：`refresh config volume=... wakeup=0 work_mode=0 curtain_type=0`。
- 数值比对：协议是否为 `55 AA 04 01 F0 B0 5B`、tone 是否为需求值、默认档位/默认值是否与需求一致。
- 是否阻断后续：若默认状态已恢复，可继续作为后续用例前置，即使协议或 tone 与需求不一致。

### 2.3 设置唤醒词
- 功能通过：进入设置入口后，目标唤醒词触发设置成功 tone，并出现 `refresh config ... wakeup=<n> ...`。
- 负例通过：
  - 设置外候选词不能形成完整唤醒链；
  - 设置窗口内普通控制词不应触发 `wakeup` 配置变化；
  - 超时后候选词不应再被当成设置项。
- 数值比对：候选词索引值、超时值、tone ID。
- 当前实机已知映射：默认 `wakeup=0`，候选词通常按列表顺序映射为 `1..7`。

### 2.4 设置工作模式
- 功能通过：进入设置入口后，目标模式词出现设置成功 tone，并出现 `refresh config ... work_mode=<n> ...`。
- 当前实机已知映射：`语音模式=0`，`滴答/嘀嗒模式=1`。
- 负例通过：
  - 设置外直接说模式词，不应出现 `work_mode` 配置变化；
  - 设置窗口内普通控制词不应触发模式切换，随后仍应能继续完成合法模式设置。
- 数值比对：切换后的控制词 tone 是否符合语音/滴答模式差异；超时值是否符合需求。

### 2.5 设置窗帘模式
- 功能通过：进入设置入口后，目标模式词出现设置成功 tone，并出现 `refresh config ... curtain_type=<n> ...`。
- 当前实机已知映射：`窗帘模式=0`，`布帘模式=1`，`纱帘模式/窗纱模式=2`。
- 负例通过：
  - 设置外直接说模式词，不应出现 `curtain_type` 配置变化；
  - 设置窗口内普通控制词不应触发模式切换，随后仍应能继续完成合法模式设置；
  - 切换后不属于当前模式的控制词即便被识别，也不能下发控制协议。
- 数值比对：tone ID、模式值、超时值。

## 3. 正负例断言分层

### 3.1 正例
- 第一层：入口可达（识别、进入设置 tone、或入口日志）。
- 第二层：状态真实变化（`wakeup/workMode/curtainType` 等配置字段变化，或动作协议下发）。
- 第三层：后置行为生效（新唤醒词可用、滴答模式控制词变成滴声、对应窗帘词可控等）。

### 3.2 负例
- 第一层：允许 ASR，但不能仅因 `keyword:` 命中就 FAIL。
- 第二层：若不该执行动作，则不得出现：
  - 禁止的配置字段变化；
  - 禁止的控制协议；
  - 禁止的完整唤醒链；
  - 禁止的成功播报。
- 第三层：需要时补一个后续正例，确认设置窗口仍在、或默认状态未被破坏。

## 4. 工具问题 / 用例问题 / DUT 问题的区分

### 4.1 更像工具问题
- 日志明显时间错位、拆行、读取滞后。
- 串口/协议口短时中断，重连或 `loglevel 4` 后恢复。
- 结果文件判 FAIL，但 `serial_raw.log` 同窗内已有完整成功证据。

### 4.2 更像用例/断言问题
- 用例语义与需求原文冲突，例如把“设置窗口内普通控制词不响应”写成“仍可响应”。
- 负例只验证“没识别”，没有验证“识别了但没执行”。
- 同一 case 同时承担“功能验证 + 数值验证 + 后置状态验证”，导致前半段失败后后半段完全没执行。

### 4.3 更像 DUT 问题
- 日志稳定、读取正常，但关键配置字段确实未变化。
- 负例出现完整误唤醒链，或不该响应的控制词真实下发了协议。
- 设置已成功但后置行为与目标模式/目标唤醒词不一致。
