# 功能点到测试用例映射草稿

## 1. 设计原则

- 一条用例只验证一个问题。
- 先验证功能，再验证数值。
- 需要依赖某个状态时，通过该用例自己的前置步骤建立状态，不依赖上一条 case 残留状态。
- 恢复出厂是独立功能点，不作为所有 case 的通用前置/收尾。
- 普通控制词干扰、超时、掉电、恢复、重启等，作为独立异常场景用例设计，不混入主功能正例。

## 2. 映射方式说明

- `功能点`：需求中要验证的具体问题。
- `建议用例`：建议拆分出的单问题 case。
- `验证重点`：该用例主要证明什么。
- `备注`：需求歧义、实测差异或后续实现提示。

## 5. 需求歧义与实现备注

- `CURTAINMODE` 需求第 5 条文字存在歧义：文案写成“进入设置窗帘模式后，若用户说纱帘/窗纱模式、布帘模式命令则不响应”，与第 4 条“进入后说模式词可切换”冲突。建议按业务逻辑理解为“进入设置后，若用户说非目标模式命令则不响应”或“进入后非模式控制词不响应”，后续需结合实测再定稿。
- `WORKMODE / WAKEWORD / CURTAINMODE` 中“设置窗口里普通控制词是否允许响应”要与实际业务分离：
  - 若需求明确“不响应”，这是主功能负例；
  - 若人工验证其可响应，则转为异常/干扰场景 case，判断的是“是否污染后续设置”。
- `SELECTOR` 与 `WAKEWORD` 的边界要明确：
  - `WAKEWORD` 关注设置入口、设置动作、设置后立即生效；
  - `SELECTOR` 关注已选择词的稳定性、误唤醒和旧词失效。
