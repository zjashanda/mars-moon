# 需求功能验证设计方法

## 1. 目的

本文档用于说明：在 `mars-moon` 当前能力基础上，拿到一份新的需求表后，如何从“需求理解”走到“验证方案设计”，再走到“结果判定与异常归因”。

本文只写方法和逻辑，不直接展开成最终测试用例。

核心原则只有四条：

1. 先验功能，再验数值。
2. 一条用例只验证一个问题。
3. 先定义证据，再写断言。
4. 功能通了但数值不对，不中断后续验证，单独记录差异。

---

## 2. 当前 skill 的验证范围

`mars-moon` 当前主要用于杜亚窗帘类离线语音项目的整套验证闭环，能力范围包括：

- 读取输入：
  - `需求文档.md`
  - `词条处理.xlsx`
  - `tone.h`
  - 可选固件 `firmware.bin`
- 生成测试资产：
  - `normalized_spec.json`
  - `tone_map.json`
  - `cases.json`
  - `deviceInfo_dooya.json`
  - 测试用例 Excel
- 实机执行：
  - 串口预检
  - 指定声卡播报
  - 日志恢复
  - 协议观测/注入
  - runner 执行
  - 结果归档
- 当前已覆盖的主要模块：
  - `BOOT`
  - `BASE`
  - `VOL`
  - `UART`
  - `POWER`
  - `FACTORY`
  - `WAKEWORD`
  - `WORKMODE`
  - `CURTAINMODE`
  - `CURTAIN`
  - `PHRASE`
  - `SELECTOR`
  - `CTRL`

因此，拿到新需求表后，第一反应不是“直接写 case”，而是先判断新需求属于已有模块、已有模块的扩展、还是需要新增 runner 能力。

## 3. 总体流程

拿到新需求表后，推荐顺序如下：

1. 先理解需求功能，不写 case。
2. 先做最小可测性验证，确认当前设备具备测试条件。
3. 把需求拆成功能点，并判断功能点之间的状态关系。
4. 给每个功能点定义“功能成立”的判断标准。
5. 再给每个功能点定义“数值一致”的判断标准。
6. 决定哪些点可自动化，哪些点只能人工验证。
7. 再把功能点落成验证方案。
8. 最后才考虑如何生成用例、如何落 runner 断言。

简化理解就是：

`需求 -> 最小可测性验证 -> 功能点 -> 状态机 -> 证据 -> 验证方案 -> 用例 -> 执行 -> 归因`

### 3.1 正式测试前的最小可测性验证

这一段是正式跑任何 case 前的强制门禁，不通过就不要继续跑全链路。

目的不是验证需求，而是先确认“设备当前可测”。

至少要确认下面两类最小逻辑：

- 语音基础链路可用
  - 设备已上电，且没有反复重启。
  - 播放默认唤醒词后，设备能被唤醒。
  - 唤醒后播放一条稳定命令词，设备能识别命令词。
  - 日志中能看到与这次动作对应的识别/唤醒证据。
  - 设备有正确的响应播报或响应音。
- 协议观测链路可用
  - 如果现场存在独立协议串口，优先使用协议串口作为协议断言依据。
  - 主动协议以协议串口实收为准，不以日志口 `send msg` 作为主断言。
  - 被动协议优先通过协议串口注入/观测链路验证，再结合日志或播报确认设备反应。

最小可测性验证推荐步骤：

1. 给 DUT 上电，并观察一段时间，确认无连续重启。
2. 打开日志口，确认能稳定读取详细日志。
3. 若有协议串口，同时打开协议口，确认波特率与收发方式正确。
4. 用指定声卡播放默认唤醒词，确认能命中唤醒。
5. 再播放一条最稳定的基础命令词，例如【打开窗帘】。
6. 同时确认：
   - 有正确识别日志
   - 有正确响应播报
   - 若该动作本应发主动协议，则协议串口上能看到对应协议

只有这一步通过，后面的需求验证才有意义。

如果这一步失败，优先归因为：

- 设备/环境问题
  - 未上电
  - 反复重启
  - 音频链路未打通
  - 日志口不可读
  - 协议口不可读
- 工具/脚本问题
  - 日志读取缓存错位
  - 协议监听口选错
  - 播放声卡选错

这类问题不应直接记成需求功能 FAIL，而应先修复最小链路。

### 3.2 烧录与烧录后健康检查

正式测试如果包含烧录，则烧录本身也必须作为测试门禁的一部分，不是“烧完就默认可测”。

烧录前的标准准备动作：

- 先区分当前环境：
  - Linux 使用 `tools/burn_bundle/linux/` 对应的烧录工具与脚本。
  - Windows 使用 `tools/burn_bundle/windows/` 对应的烧录工具与脚本。
- 将待烧录固件复制到当前实际执行的烧录工具目录。
- 复制前先删除旧的 `app.bin`，避免目录中残留多个固件。
- 将本次固件统一重命名为 `app.bin`，确保当前烧录目录中固件唯一。

进入烧录模式的控制时序必须固定为：

1. 断电
2. 进 boot
3. 上电
4. 下 boot

只有这四个动作全部成功、且中间无异常，才允许进入烧录模式。

如果其中任何一步失败：

- 不允许从中间步骤续接。
- 必须回到起点，重新完整执行一遍“断电 -> 进 boot -> 上电 -> 下 boot”。

烧录完成后，还不能直接进入测试，必须追加健康检查：

1. 恢复正常启动时序。
2. 连接日志串口。
3. 上电后持续观察至少 `20s`。
4. 检查日志中是否存在自动重启、重复启动、异常复位或其他明显运行异常。

烧录后健康检查结论规则：

- 若 `20s` 内启动稳定、无异常重启，可进入后续最小可测性验证。
- 若存在自动重启、反复启动、日志异常中断，则停止测试并上报设备/固件异常。

注意顺序：

- 烧录后健康检查通过，不代表可以直接跑用例。
- 仍必须继续执行第 `3.1` 节的最小可测性验证。
- 只有“烧录后健康检查通过 + 最小可测性验证通过”两者同时成立，才允许进入正式测试。

## 4. 拿到新需求表后的分析模板

这一部分的目标不是产出用例，而是先把需求读成“可验证对象”。

### 4.1 先提取基础信息

先把以下信息从需求表中抽出来：

- 功能名称是什么
- 功能属于哪个模块
- 入口动作是什么
- 目标状态是什么
- 成功后行为应该变成什么
- 是否有默认值
- 是否有边界值
- 是否有超时
- 是否涉及掉电/重启/恢复出厂
- 是否涉及协议
- 是否涉及播报 ID / tone 文案
- 是否涉及日志字段
- 是否有负例限制
- 是否有依赖前置状态

如果一个需求点抽不出这几项，先不要写方案，先标记为“需求信息不完整”。

### 4.2 把需求按功能类型分类

建议优先分成以下几类：

- 基础会话类
  - 唤醒
  - 进入交互态
  - 超时退出
  - 退出后重唤醒
- 设置类
  - 设置唤醒词
  - 设置工作模式
  - 设置窗帘模式
- 控制类
  - 打开/关闭/停止
  - 同义词
  - 词组等价
- 配置类
  - 默认值
  - 档位数
  - 超时值
  - 协议波特率
  - 日志波特率
- 状态保持类
  - 掉电保持
  - 恢复出厂
- 交互窗口类
  - 设置窗口是否超时
  - 窗口内普通命令是否干扰设置
  - 配对窗口关闭后是否仍响应
- 观测链路类
  - 日志口
  - 协议口
  - 被动协议注入
- 人工确认类
  - 欢迎语
  - 静态配置
  - 自动化无法稳定取证的项目

分类的目的，是让后面的验证方案直接复用同类逻辑，而不是每次从零开始。

### 4.3 判断它是“功能需求”还是“数值需求”

很多需求其实混了两层东西：

- 功能需求
  - 能不能做
  - 会不会生效
  - 会不会退出
  - 会不会误响应
- 数值需求
  - 协议是什么
  - 播报 ID 是什么
  - 超时时间是多少
  - 默认值是多少
  - 步进数是多少

必须把两层拆开。

例如“恢复出厂后默认唤醒词恢复，协议为 XXX，播报为 XXX”：

- 功能需求：恢复出厂后默认唤醒词恢复
- 数值需求：恢复出厂命令的协议和值

如果不拆开，最终就会出现“功能明明正常，但因为协议不一致整条链路都被判 FAIL”的问题。

### 4.4 识别状态依赖

每个功能点都要先判断是不是依赖状态。

常见状态依赖有：

- 是否已唤醒
- 是否在设置窗口内
- 当前工作模式是什么
- 当前窗帘模式是什么
- 当前唤醒词是什么
- 当前音量档位是什么
- 当前是否刚恢复出厂
- 当前是否刚掉电重启
- 当前是否处于配对窗口

只要有状态依赖，就不能让用例依赖上一条 case 的残留状态，而要在该功能点自己的前置步骤里自己把状态建立出来。

### 4.5 识别当前 skill 能否直接覆盖

分析完需求点后，马上做一次映射：

- 已有模块能直接覆盖
  - 只需要补词条、协议、tone、默认值、超时值
- 已有模块能覆盖，但断言口径需要调整
  - 例如：功能先于数值
  - 例如：主动协议只认协议口，不认日志口
- 现有能力不能覆盖
  - 需要新增日志字段解析
  - 需要新增协议注入动作
  - 需要新增特殊等待/观察逻辑
  - 需要标成人工项

如果这个判断不先做，后面很容易把“需求问题”写成“runner 问题”。

## 5. 功能点到验证方案的设计逻辑

这一部分是核心。

目标不是写出用例编号，而是回答：

- 这个功能点到底应该怎么验证
- 什么结果才算功能通过
- 正例、负例、异常场景怎么拆
- 证据应该采什么
- 哪些动作是前置，哪些动作才是主体

### 5.1 每个功能点必须先写成一个验证卡片

每个功能点，在写方案前先整理成下面这张“验证卡片”：

1. 功能名称
2. 所属模块
3. 功能目的
4. 入口动作
5. 前置状态
6. 目标状态
7. 成功后可观察行为
8. 禁止出现的行为
9. 必要观测证据
10. 辅助观测证据
11. 功能通过标准
12. 数值一致标准
13. 反例验证点
14. 异常场景验证点
15. 自动化/人工化结论

只有这张卡片写清楚了，后面才值得写用例。

### 5.2 先定义“功能通过标准”

功能通过标准是第一层判断，必须独立存在。

推荐的判断方式：

- 入口动作被正确识别
- 且目标状态已经发生变化
- 且变化后的行为符合目标功能

常见功能通过的闭环写法：

- 设置类功能
  - 进入设置 -> 选择目标项 -> 设置后行为变化
- 模式类功能
  - 设置前不生效 -> 进入设置后切换 -> 切换后目标模式行为成立
- 控制类功能
  - 唤醒 -> 说控制词 -> 实际动作反馈成立
- 超时类功能
  - 进入窗口 -> 等待 -> 窗口退出 -> 退出后行为变化
- 保持类功能
  - 先改离默认 -> 掉电/恢复 -> 再看最终状态

功能通过标准里不要直接写协议值、tone 值；这些属于第二层。

### 5.3 再定义“数值一致标准”

在功能已经成立的前提下，再检查数值是否一致。

数值层通常包括：

- 协议值
- 播报 ID
- 播报文案
- 超时时长
- 默认值
- 步数
- 档位数
- 日志字段值

写法建议是：

- 功能成立
- 实测值 = ?
- 需求值 = ?
- 是否一致 = 是/否

这样即使数值不一致，也不会把“功能正常”误写成“功能失败”。

### 5.4 为每个功能点拆正例、负例、异常场景

一个功能点至少从三个方向考虑：

#### A. 正例

验证“该成立时是否成立”。

关注：

- 最短路径证明功能可用
- 成功证据是否完整
- 是否需要后置行为验证

#### B. 负例

验证“本不该成立时是否真的不成立”。

常见负例：

- 未唤醒直接说命令不响应
- 未进入设置时模式词不生效
- 未设置的候选唤醒词不应唤醒
- 非当前模式的专有控制词不应响应
- 窗口外注入协议不应生效

负例必须独立设计，不要混在正例尾部顺手一验。

#### C. 异常/边界场景

验证“功能在特殊条件下是否仍符合业务规则”。

常见异常场景：

- 设置窗口超时
- 配对窗口关闭后再次注入
- 掉电后恢复
- 恢复出厂后恢复默认
- 普通控制词是否干扰设置
- 重启后状态是否恢复

异常场景的目的不是再验证一次正例，而是确认状态机边界。

### 5.5 前置步骤和主体验证必须分开

这点必须严格执行。

前置步骤的职责只有一个：

- 把设备摆到正确的验证状态

主体验证的职责只有一个：

- 判断这一个功能点是否成立

例如“恢复出厂后布帘词失效”：

- 前置：
  - 进入窗帘模式设置
  - 切到布帘模式
  - 确认布帘模式已生效
  - 执行恢复出厂
- 主体：
  - 说 `打开布帘`
  - 判断是否仍响应

不要把“切模式成功”“恢复出厂成功”“布帘词失效”三个判断揉成一个大结论。

### 5.6 证据分层：必要证据 vs 辅助证据

当前 `mars-moon` 可用证据很多，但不能全都当硬断言。

建议分两层：

#### 必要证据

能直接证明功能是否成立的证据。

例如：

- `wakeKw` / `asrKw`
- `wakeReady`
- `playId`
- 协议串口实收
- 被动协议注入后的设备响应
- 超时退出后的行为变化

#### 辅助证据

能帮助分析原因，但不应该一上来就直接判死的证据。

例如：

- `configSaved`
- `configRefresh`
- 某些状态日志
- 日志中的附加字段

典型原则：

- `configSaved` 可以辅助判断“设置是否落盘”
- 但不能把所有设置类用例都写成“没有 `configSaved` 就直接 FAIL”

协议相关场景需要额外补一条优先级规则：

- 只要现场存在独立协议串口，协议串口证据优先级高于日志口协议文本。
- 主动协议：
  - 必要证据优先定义为协议串口实收帧。
  - 日志里的 `send msg` 只能作为辅助证据。
- 被动协议：
  - 必要证据优先定义为协议串口注入动作成立，以及设备对注入后的行为反馈成立。
  - 日志里的 `recv msg`、播报 ID、窗口变化作为辅助交叉印证。

这样可以避免“日志里看见了协议文本，但真正协议口并没有收发”的假通过。

### 5.7 按功能类型套用设计模式

为了提高稳定性，可以按功能类型套固定逻辑。

#### 5.7.1 入口可达型

适用：

- 恢复出厂入口
- 设置工作模式入口
- 设置窗帘模式入口
- 配对入口

设计逻辑：

- 验证入口词是否被识别
- 验证设备是否进入对应窗口/状态
- 不急着同时验证窗口内所有后续行为

重点：

- “入口可达”是一条独立验证，不和“入口后的所有功能”绑定

#### 5.7.2 默认值型

适用：

- 默认工作模式
- 默认窗帘模式
- 默认音量
- 默认唤醒词

设计逻辑：

- 把设备放到默认状态
- 选一条最稳定的观测动作验证默认态行为

重点：

- 默认值型验证不应混入切换、恢复、掉电等动作

#### 5.7.3 设置后立即生效型

适用：

- 设置唤醒词
- 切换工作模式
- 切换窗帘模式

设计逻辑：

- 进入设置窗口
- 说目标设置词
- 立刻验证设置后的目标行为

重点：

- 必须有“设置后行为变化”这一步
- 不能只看设置播报就算通过

#### 5.7.4 设置窗口抗干扰型

适用：

- 设置唤醒词窗口
- 设置工作模式窗口
- 设置窗帘模式窗口

设计逻辑：

- 进入设置窗口
- 插入一条普通控制命令
- 再继续完成原设置
- 判断普通命令是否污染设置流程

重点：

- 验证的是“干扰后是否还能继续完成设置”
- 不是默认把普通命令定义成无效词

#### 5.7.5 超时值与超时退出型

适用：

- 会话窗口超时
- 设置窗口超时
- 配对窗口超时

设计逻辑：

- 先验证“会不会退出”
- 再验证“多久退出”

重点：

- 超时功能成立与超时数值一致是两层结论

#### 5.7.6 模式互斥型

适用：

- 布帘模式 / 纱帘模式 / 窗纱模式
- 当前候选唤醒词 vs 非当前候选唤醒词

设计逻辑：

- 先把目标模式/目标选择项建立好
- 再验证当前目标正例是否成立
- 再验证其他项负例是否不成立

重点：

- 正例和负例拆开设计
- 不要一条 case 同时验证所有互斥项

#### 5.7.7 掉电保持型

适用：

- 唤醒词掉电保持
- 音量掉电保持

设计逻辑：

- 先把目标状态改离默认值
- 确认设置动作已足够成立
- 再掉电
- 再验证掉电后的最终状态

重点：

- 如果掉电前连设置都没确认成立，就不能直接下掉电结论

#### 5.7.8 恢复出厂型

适用：

- 唤醒词恢复默认
- 音量恢复默认
- 工作模式恢复默认
- 窗帘模式恢复默认

设计逻辑：

- 先人为改成非默认
- 确认非默认状态已生效
- 执行恢复出厂
- 再单独验证默认状态恢复

重点：

- 恢复出厂是独立功能，不作为所有用例的通用前置或收尾

#### 5.7.9 协议观测型

适用：

- 主动协议发送
- 协议口波特率
- 日志关键字段

设计逻辑：

- 先确认观测链路本身可用
- 再选稳定命令做协议观测
- 协议口实收优先于日志口 `send msg`

重点：

- 协议链路问题不能直接等价于功能没实现
- 要区分“功能真失败”和“观测链路拿错值”

#### 5.7.10 被动注入窗口型

适用：

- 遥控器配对成功/失败
- 窗口外注入无效

设计逻辑：

- 先进入目标窗口
- 在窗口内注入
- 验证响应
- 窗口关闭后再次注入
- 验证不再响应

重点：

- 窗口内成立和窗口外失效是两条独立验证

### 5.8 当前 skill 下常见的错误设计方式

以下设计方式应避免：

- 一条用例同时验证入口、设置、正例、负例、超时、恢复
- 默认把所有日志字段都当硬断言
- 只看到识别文本就判“误唤醒”
- 前置恢复动作污染主体验证结果
- 还没确认功能是否成立，就先按协议/tone 值把结论判死
- 明明功能已经正常，却因为一个数值不同而停止后续回归

## 6. 按模块展开的测试方案草稿

这一章把前面的通用方法，进一步落到当前 `mars-moon` 已覆盖模块上。

这些内容仍然是“验证方案”，不是最终测试用例；其作用是帮助后续在写 case 时，先明确每个模块该怎么设计验证逻辑。

### 6.1 BOOT 上电欢迎语

- 功能说明
  - 验证设备上电/重启后是否播报欢迎语，系统是否回到可测试状态。
- 正向验证
  - 触发一次硬重启或上下电。
  - 观察欢迎语播报、日志恢复、设备重新进入待测状态。
- 通过标准
  - 出现欢迎语对应播报或日志证据。
  - 重启后日志链路恢复，后续可继续执行语音功能。
- 反例验证
  - 非重启场景下不应重复播报欢迎语。
- 异常场景
  - 只重启但日志没恢复：功能不算完成。
  - 有欢迎语但系统无法继续交互：判启动链路异常。

### 6.2 BASE 基础会话状态机

- 功能说明
  - 验证唤醒后是否进入识别窗口，超时是否退出，退出后是否需要重新唤醒。
- 正向验证
  - 播放默认唤醒词。
  - 观察进入识别态。
  - 等待超时退出。
  - 超时后重新唤醒再执行一条普通命令。
- 通过标准
  - 唤醒成功后进入交互态。
  - 超时后出现退出证据。
  - 退出后直接说命令不响应，重新唤醒后恢复响应。
- 反例验证
  - 未唤醒直接说命令不应响应。
- 异常场景
  - 超时不退出或退出后仍能直接执行命令：会话状态机异常。
  - 唤醒证据和超时证据总是错位：优先排查脚本观察窗口。

### 6.3 VOL 音量功能

- 功能说明
  - 验证默认音量、音量增减、最小/最大边界提示及步进次数。
- 正向验证
  - 从默认音量开始，逐步增大到边界，再逐步减小到边界。
  - 验证每一步普通反馈和边界反馈。
- 通过标准
  - 默认音量符合需求或有明确实测值。
  - 普通调节有普通反馈。
  - 达到最小/最大时出现边界提示。
  - 到达边界前的步数与需求一致或明确记录差异。
- 反例验证
  - 已到最大仍继续增大，不应再出现普通增大反馈。
  - 已到最小仍继续减小，不应再出现普通减小反馈。
- 异常场景
  - 音量起点和需求不一致但调节逻辑正常：功能通过，数值差异单独记录。
  - 边界提示缺失或步进逻辑错误：功能异常。

### 6.4 UART 日志/协议观测链路

- 功能说明
  - 验证语音动作后日志口、协议口是否有对应输出。
- 正向验证
  - 选一条稳定控制词，例如打开窗帘。
  - 观察日志中的识别、播报，以及协议口上的实际协议。
- 通过标准
  - 日志口可读。
  - 若现场存在协议串口，则协议口有对应协议。
  - 动作结果和日志/协议可互相印证。
- 反例验证
  - 非法词或未唤醒状态下，不应出现控制协议。
- 异常场景
  - 功能听起来执行了，但日志或协议缺失：先归观测链路问题，不直接否定功能。
  - 串口断连或波特率不匹配：先修复环境，再测功能。
  - 有独立协议串口时，日志中的 `send msg` / `recv msg` 不能替代协议串口断言。

- 断言优先级
  - 主动协议：
    - 第一断言来源：协议串口实收帧。
    - 第二断言来源：日志 `send msg`。
  - 被动协议：
    - 第一断言来源：协议串口注入是否成功、设备是否产生预期反应。
    - 第二断言来源：日志 `recv msg`、播报、窗口状态变化。

- 最小链路门禁
  - 正式测试前，必须先用“唤醒词 + 稳定控制词”做一次最小链路验证。
  - 只有同时确认日志可读、播报正确、主动协议可在协议口上观测到，才进入后续大批量用例执行。

### 6.5 POWER 掉电保持

- 功能说明
  - 验证掉电后哪些配置保留，哪些恢复默认。
- 正向验证
  - 先把目标配置改成非默认值。
  - 执行掉电/上电。
  - 验证配置是否按需求保留或恢复。
- 通过标准
  - 唤醒词、音量等状态变化符合需求配置。
- 反例验证
  - 本该不保留的配置不应被保留。
  - 本该保留的配置不应恢复默认。
- 异常场景
  - 若设置动作本身都未成功，不进入掉电结论。
  - 若掉电前未等到保存落盘证据，结果需要标注“掉电时机可能影响结论”。

### 6.6 FACTORY 恢复出厂

- 功能说明
  - 验证恢复出厂后默认唤醒词、默认音量、默认工作模式、默认窗帘模式是否恢复。
- 正向验证
  - 先人工把目标状态改离默认值。
  - 执行恢复出厂模式。
  - 分别验证默认状态是否恢复。
- 通过标准
  - 默认唤醒词恢复。
  - 默认音量恢复。
  - 默认工作模式恢复。
  - 默认窗帘模式恢复。
- 反例验证
  - 已改过的状态不应继续保留旧值。
- 异常场景
  - 恢复出厂功能已成立但协议/TONE 不一致：记为功能通过、数值差异，不阻塞后续依赖。

### 6.7 WAKEWORD 设置唤醒词

- 功能说明
  - 验证默认唤醒词始终有效，候选唤醒词只能在设置后生效，设置窗口超时后退出。
- 正向验证
  - 默认唤醒词可唤醒。
  - 进入设置唤醒词模式后，设置某个候选唤醒词。
  - 设置后该候选词立即生效。
- 通过标准
  - 默认词始终可唤醒。
  - 候选词在设置后可唤醒。
  - 设置动作和设置后行为变化一致。
- 反例验证
  - 未进入设置模式时，候选词不应直接唤醒。
  - 设置为 A 后，未设置的 B/C/D 不应误唤醒。
- 异常场景
  - 设置窗口内普通控制词如果业务允许响应，则验证“普通命令可执行 + 后续设置仍可完成”，不能当静默负例。
  - 设置成功播报到了，但新词不生效：判功能未完成。
  - 新词生效但日志/播报和需求不一致：判功能通过、数值差异。

### 6.8 WORKMODE 设置工作模式

- 功能说明
  - 验证语音模式/嘀嗒模式切换规则，以及不同模式下反馈口径是否变化。
- 正向验证
  - 进入设置工作模式。
  - 分别切到语音模式和嘀嗒模式。
  - 切换后用一条稳定命令验证反馈差异。
- 通过标准
  - 进入设置前模式词不生效。
  - 进入设置后模式词可切换成功。
  - 切换后同一命令的反馈口径发生预期变化。
- 反例验证
  - 未进入设置态时，说模式词不应响应。
  - 切到某模式后，不应仍保留另一模式的反馈口径。
- 异常场景
  - 模式切换成功但播报 ID 与需求不同：功能通过、数值差异。
  - 设置窗口内普通控制词若可执行，需验证不会阻断后续模式设置。

### 6.9 CURTAINMODE 设置窗帘模式

- 功能说明
  - 验证窗帘模式、纱帘/窗纱模式、布帘模式切换，以及模式生效后的命令词范围。
- 正向验证
  - 进入设置窗帘模式。
  - 分别切到目标模式。
  - 切换后验证该模式下专有词有效。
- 通过标准
  - 进入设置前，模式词不生效。
  - 进入设置后，目标模式可切换成功。
  - 切换后对应模式专有词有效。
- 反例验证
  - 切到纱帘/窗纱模式后，布帘专有词不应有效。
  - 切到布帘模式后，纱帘/窗纱专有词不应有效。
- 异常场景
  - 通用窗帘词不能粗暴拿来做模式互斥负例；它应按实际业务单独验证。
  - 设置成功但模式词集合未切换：功能异常。

### 6.10 CURTAIN 窗帘控制命令

- 功能说明
  - 验证开/关/停等控制命令在不同模式下是否映射到正确行为。
- 正向验证
  - 在默认模式和各专有模式下执行对应控制词。
  - 观察播报和协议发送。
- 通过标准
  - 对应命令可被正确识别。
  - 有正确播报/协议或至少功能行为成立。
- 反例验证
  - 非当前模式下的专有控制词不应生效。
  - 非法词不应产生协议动作。
- 异常场景
  - 同义词识别与主词不一致：转 `PHRASE` 模块继续分析。
  - 协议不一致但动作行为正确：功能通过，协议差异记录。

### 6.11 PHRASE 词条/同义词识别

- 功能说明
  - 验证同义词、等价短语是否能被识别为同类合法意图。
- 正向验证
  - 以同一意图的多个表达逐条验证。
  - 比较识别文本是否落在可接受集合内。
- 通过标准
  - 同组词条都能被正确接受。
- 反例验证
  - 跨意图词条不应被误识别成当前目标意图。
- 异常场景
  - 若功能正常但 ASR 文本与词表写法不同，只要属于等价表达，应判通过并记录识别差异。

### 6.12 SELECTOR 定向/选择性唤醒

- 功能说明
  - 验证设置某个候选唤醒词后，仅该词可稳定唤醒，其他候选词不应误唤醒。
- 正向验证
  - 设置当前候选唤醒词为 A。
  - 连续多次用 A 唤醒。
- 通过标准
  - A 可稳定唤醒。
  - 默认唤醒词仍可唤醒（若需求要求保留）。
- 反例验证
  - 未设置的候选词 B 不应误唤醒。
  - 切换到 B 后，旧词 A 不应继续有效。
- 异常场景
  - A/B 发音相近时，误唤醒必须按真正唤醒证据判定，不能仅凭识别文本浮现判错。

### 6.13 CTRL 遥控器配对/清除

- 功能说明
  - 验证进入配对窗口、窗口内成功/失败被动播报、窗口外注入无效、清除遥控器等链路。
- 正向验证
  - 先进入配对窗口。
  - 在窗口内注入成功协议、失败协议。
  - 执行清除遥控器命令。
- 通过标准
  - 能进入正确窗口。
  - 窗口内注入成功/失败协议后有对应被动播报。
  - 清除动作有对应反馈。
- 反例验证
  - 窗口外注入成功/失败协议不应生效。
- 异常场景
  - 入口协议和需求不一致，但窗口功能成立：功能通过，协议差异记录。

### 6.14 本章与前文的关系

- 本章只告诉你“每个模块该怎么设计验证方案”。
- 真正写用例时，仍应遵守第 `5` 章的方法：
  - 一条用例只验证一个问题
  - 前置与主体分开
  - 必要证据与辅助证据分开
  - 正例、负例、异常场景拆开

## 7. 结果记录与归因模板

执行完成后，结果建议按“三层输出”组织。

### 7.1 第一层：功能结论

先回答功能是否成立：

- PASS：功能成立
- FAIL：功能未成立
- PARTIAL：功能成立，但部分数值不一致
- DRY_RUN / NO_METHOD：当前未自动执行

### 7.2 第二层：数值差异

如果功能成立，再单列：

- 需求值
- 实测值
- 是否一致
- 差异说明

推荐记录格式：

- 功能：是否正常
- 协议：需求 / 实测
- 播报：需求 / 实测
- 超时：需求 / 实测
- 日志：需求 / 实测

### 7.3 第三层：异常归因

归因时优先按下面顺序判断：

1. 环境问题
   - 设备没上电
   - 日志口断
   - 协议口断
   - 声卡没播出
   - 控制口不可用
2. 观测/runner 问题
   - 观察窗口不对
   - 前一条协议污染当前动作
   - 负例判定过严或过松
   - 收口逻辑错误
3. 用例设计问题
   - 多功能点耦合
   - 前置和主体混淆
   - 需求理解错位
4. 固件功能问题
   - 功能真的不通
   - 不该响应时却响应
   - 状态切换不生效
5. 需求/词表/tone 差异
   - 功能通了，但值和需求不一致

### 7.4 是否继续后续测试

结果出来后再决定是否继续：

- 如果是功能差异但链路可观测：继续后续测试
- 如果是数值差异：继续后续测试
- 如果是日志/协议完全不可观测：先停，修环境
- 如果是连续唤醒失败、设备异常重启：按停测规则处理

## 8. 最终落地建议

当一份新需求表进入 `mars-moon` 时，推荐实际执行顺序如下：

1. 先按本文档完成需求拆解，不写 case。
2. 产出“功能点验证卡片”。
3. 确认每个功能点的：
   - 功能通过标准
   - 数值一致标准
   - 正例/负例/异常场景
   - 必要证据/辅助证据
4. 再决定是否需要改：
   - `tools/dooya_spec_builder.py`
   - `tools/dooya_case_builder.py`
   - `tools/dooya_voice_runner.py`
5. 先做：
   - `prepare`
   - `build`
   - `run --dry-run`
6. 若涉及新固件，先区分 Linux / Windows 烧录环境，准备当前平台烧录工具目录。
7. 将待烧录固件复制到烧录目录，删除旧 `app.bin` 后再将新固件统一命名为 `app.bin`。
8. 执行完整烧录进入时序：
   - 断电
   - 进 boot
   - 上电
   - 下 boot
   - 四步任一步失败，都必须整套重来
9. 烧录完成后连接日志串口并上电观察 `20s`：
   - 无异常重启，进入下一步
   - 有自动重启或明显异常，立即停止测试并上报
10. 再执行最小可测性验证：
   - 默认唤醒词可唤醒
   - 基础命令词可识别
   - 有对应日志
   - 有对应响应播报
   - 若存在协议串口，主动/被动协议优先走协议串口校验
11. 只有烧录后健康检查和最小可测性验证都通过，才进入正式 `run`。
12. 最终按“功能结论 + 数值差异 + 异常归因”出结果。

如果严格按这个顺序走，测试会更稳定，也更容易区分：

- 是功能没实现
- 还是需求值写错
- 还是 runner 断言错了
- 还是工位环境有问题
