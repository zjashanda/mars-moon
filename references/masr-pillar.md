# masr-pillar

## 1. 文档目的

本文档把“当前项目需求内容提炼”和“对应测试方案设计示例”合在同一份文档里，作为 `mars-moon` 的示范样板。

目标不是替代原始需求，而是让后续其他 skill 在读取这份文档后，能快速理解：

- 当前这类杜亚离线语音项目到底要测什么
- 需求内容应该怎么抽取成可验证对象
- 烧录、预检、停测门禁应该怎么写
- 功能点应该如何转成测试方案
- 功能正常但数值不一致时，应如何继续测试并做归因

当前示例主要来源于：

- `work/normalized_spec.json`
- `generated/deviceInfo_dooya.json`
- `work/tone_map.json`
- `references/需求功能验证设计方法.md`

## 2. 当前需求内容提炼示例

### 2.1 项目信息

- 项目名称：`CSK5062杜亚窗帘`
- 分支名称：`csk5062_dooya_curtain`
- 芯片型号：`CSK5062`
- 固件版本：`VCC.03.02.00.24`

### 2.2 基础配置要求

- 唤醒时长：`20s`
- 音量档位：`4`
- 默认音量：`3`
- 最小音量边界提示：`TONE_ID_5 / 音量已最小`
- 最大音量边界提示：`TONE_ID_4 / 音量已最大`
- 协议串口：`UART1 / 9600`
- 日志串口：`UART0 / 115200`
- 唤醒词掉电保存：`否`
- 音量掉电保存：`否`
- 默认工作模式：`语音模式`
- 默认窗帘模式：`窗帘模式`

在当前 `mars-moon` 实机示例里，对应口径被落成：

- 日志口：`/dev/ttyACM0 @ 115200`
- 协议口：`/dev/ttyACM2 @ 9600`

### 2.3 当前需求中的核心功能点

#### A. 基础会话

- 默认唤醒词：`你好杜亚`
- 候选唤醒词：
  - `客厅窗帘`
  - `餐厅窗帘`
  - `阳台窗帘`
  - `卧室窗帘`
  - `左边窗帘`
  - `中间窗帘`
  - `右边窗帘`
- 默认唤醒词一直生效
- 唤醒后进入识别态
- 超时值要求为 `20s`
- 需要先唤醒再播普通命令词

#### B. 音量功能

- 音量档位总数：`4`
- 默认音量档位：`3`
- 普通调节反馈：`TONE_ID_1 / 好的`
- 最大边界反馈：`TONE_ID_4 / 音量已最大`
- 最小边界反馈：`TONE_ID_5 / 音量已最小`
- 掉电后恢复默认
- 恢复出厂后恢复默认

#### C. 恢复出厂

- 入口词：`恢复出厂模式`
- 当前需求口径中的主动协议：`55 AA 04 01 F0 B0 5B`
- 当前需求口径中的反馈：`TONE_ID_1 / 好的`
- 恢复项：
  - 默认唤醒词
  - 默认音量
  - 默认工作模式
  - 默认窗帘模式

#### D. 设置唤醒词

- 入口词：`设置唤醒词`
- 入口反馈：`TONE_ID_9 / 进入设置模式`
- 默认唤醒词：`你好杜亚`
- 候选唤醒词默认不生效，进入设置后才可切换
- 设置成功后，新唤醒词立即生效并退出设置态
- 设置超时后，自动退出设置态
- 设置窗口内需继续验证：
  - 新词是否生效
  - 默认词是否仍可用
  - 其他未设置候选词是否误唤醒
  - 超时是否仍为 `20s`

#### E. 设置工作模式

- 入口词：`设置工作模式`
- 入口反馈：`TONE_ID_9 / 进入设置模式`
- 模式候选：
  - `语音模式` -> `TONE_ID_1 / 好的`
  - `滴答模式` -> `TONE_ID_15 / 滴声`
- 默认模式：`语音模式`
- 未进入设置态时，直接说模式词不应生效
- 进入设置态后，切换成功应退出设置态
- 设置超时后，自动退出设置态

#### F. 设置窗帘模式

- 入口词：`设置窗帘模式`
- 入口反馈：`TONE_ID_9 / 进入设置模式`
- 默认模式：`窗帘模式`
- 候选模式：
  - `纱帘模式`
  - `窗纱模式`
  - `布帘模式`
- 当前词表中候选模式设置成功反馈：`TONE_ID_10 / 设置成功`
- 未进入设置态时，模式词不应生效
- 进入设置态后，目标模式设置成功应退出设置态
- 设置超时后，自动退出设置态

#### G. 窗帘控制与词组等价

当前默认窗帘控制主协议：

- `打开窗帘 / 窗帘打开` -> `55 AA 05 01 01 01 EA A9`
- `窗帘关闭 / 关闭窗帘` -> `55 AA 05 01 01 02 EA A3`
- `停止窗帘 / 窗帘停止` -> `55 AA 05 01 01 03 6A A6`

当前需求还隐含三类验证：

- 同义词是否等价
- 专有模式词是否只在对应模式下有效
- 非当前模式词是否不响应

#### H. 定向/选择性唤醒

- 设置某个候选唤醒词后，该词应可稳定唤醒
- 默认唤醒词仍应可用
- 其他未设置候选词不应误唤醒
- 切换到新候选词后，旧候选词应失效

#### I. 遥控器配对/清除

- 入口词：`配对遥控器`
- 主动协议：`55 AA 05 01 12 00 80 AF`
- 入口反馈：`TONE_ID_6 / 开始配对`
- 被动成功协议：`55 AA 05 03 06 01 F8 81`
- 被动失败协议：`55 AA 05 03 06 00 78 84`
- 成功反馈：`TONE_ID_7 / 配对成功`
- 失败反馈：`TONE_ID_8 / 配对失败`
- 配对窗口超时：`20s`
- 清除遥控器：`清除遥控器` -> `55 AA 05 01 16 00 18 AC`

### 2.4 当前需求补充逻辑中的关键测试思想

当前项目的补充说明里，实际已经给出了几个非常重要的测试逻辑，这些应被视为方案设计的一部分：

- 唤醒时长不能只看需求，要实测等待超时来验证数值
- 音量档位不能只看默认值，要通过连续调节确认档位范围和边界
- 默认音量应通过烧录后或恢复出厂后重新验证
- 协议串口的职责是看真实协议输出，不应只依赖日志文本
- 掉电保存类必须先把值改离默认，再硬断电验证
- 恢复出厂类必须先改状态，再恢复出厂，再看默认值是否回来
- 默认唤醒词是公共基础能力，后续很多功能验证都可依赖它进入识别态
- 三个设置功能都存在超时退出，不应只测设置成功，不测超时退出

### 2.5 当前需求的解释原则

这里必须保留一个核心原则：

- 需求文档、补充说明、词表、tone 表和固件实装值可能并不完全一致。
- 测试时先确认“功能是否成立”，再确认“数值是否一致”。
- 如果功能成立但协议、播报 ID、文案、超时数值与需求不同，应继续后续测试，并把差异单独记录。
- 如果功能本身不成立，才判为真正的功能 FAIL。

## 3. 对应测试方案示例

### 3.1 总体设计原则

- 一条用例只验证一个问题
- 前置步骤只负责建立状态，不负责输出主体结论
- 主体验证只负责判断当前这个功能点是否成立
- 先定义必要证据，再定义辅助证据
- 先验功能，再验数值
- 功能通过但数值不符，不中断后续测试

### 3.2 烧录前后的门禁设计

#### 3.2.1 烧录前准备

- 先区分平台：
  - Linux：使用 `tools/burn_bundle/linux/`
  - Windows：使用 `tools/burn_bundle/windows/`
- 将当前待烧录固件复制到实际烧录目录
- 删除旧 `app.bin`
- 将新固件统一命名为 `app.bin`
- 保证烧录目录内固件唯一

#### 3.2.2 进入烧录模式的控制时序

固定执行：

1. 断电
2. 进 boot
3. 上电
4. 下 boot

规则：

- 这四步都成功，才允许进入烧录模式
- 只要任一步失败，就必须整套重新执行
- 不允许从中间步骤接着做

#### 3.2.3 烧录后健康检查

- 烧录完成后，恢复正常启动
- 连接日志串口
- 上电后观察 `20s`
- 检查是否存在：
  - 自动重启
  - 反复启动
  - 日志中断
  - 明显异常复位

结论规则：

- `20s` 内稳定：可进入最小可测性验证
- `20s` 内有自动重启或明显异常：停止测试并上报问题

### 3.3 正式测试前的最小可测性验证

这是正式执行所有 case 前的第二道硬门禁。

推荐只跑一条最稳定短链路：

1. 给设备上电
2. 打开日志口
3. 若有协议口，同时打开协议口
4. 用指定声卡播放默认唤醒词 `你好杜亚`
5. 再播放稳定控制词，例如 `打开窗帘`
6. 同时确认：
   - 设备无反复重启
   - 唤醒成功
   - 命令词识别成功
   - 日志中有对应识别/状态证据
   - 有对应响应播报
   - 若该动作应发主动协议，则协议串口有实收协议

如果这一步失败：

- 直接停止正式测试
- 不进入大批量用例执行
- 优先归因为设备/环境/工具问题，而不是需求 FAIL

### 3.4 协议断言优先级方案

只要现场存在独立协议串口，就统一使用下面的优先级：

#### 主动协议

- 第一证据：协议串口实收帧
- 第二证据：日志中的 `send msg`
- 第三证据：播报或行为结果

#### 被动协议

- 第一证据：协议串口注入动作成立
- 第二证据：设备对该注入的行为反馈成立
- 第三证据：日志中的 `recv msg`、播报 ID、窗口状态变化

结论：

- 不能拿日志中的 `send msg` / `recv msg` 去替代协议串口本身
- 协议串口只要可用，就优先用协议串口校验主动/被动协议

### 3.5 按模块设计测试方案示例

#### A. BOOT

- 测什么：上电欢迎语、上电后是否可进入待测状态
- 怎么测：重启或上下电一次，观察欢迎语和日志恢复
- 通过标准：欢迎语成立，后续链路可继续测试
- 异常判断：有欢迎语但系统无法继续交互，仍算启动异常

#### B. BASE

- 测什么：默认唤醒、进入交互态、超时退出、退出后需重唤醒
- 怎么测：默认唤醒 -> 等待超时 -> 未唤醒说命令 -> 再次唤醒说命令
- 通过标准：超时前后状态切换符合预期，未唤醒不响应，重唤醒后恢复
- 数值验证：超时是否约 `20s`

#### C. VOL

- 测什么：默认音量、音量步进、上下边界、掉电恢复、恢复出厂恢复
- 怎么测：连续增减音量，数普通步数和边界提示，再结合掉电/恢复出厂验证
- 通过标准：调节逻辑正常，边界反馈正确
- 数值验证：默认档位是否为 `3`、总档位是否为 `4`

#### D. UART

- 测什么：日志口是否可读、协议口是否有真实协议
- 怎么测：执行一条稳定控制词并同时观察日志和协议口
- 通过标准：日志、播报、协议三者互相印证
- 特别规则：有独立协议口时，协议口优先级最高

#### E. POWER

- 测什么：掉电后唤醒词和音量是否恢复默认
- 怎么测：先改离默认值，再硬断电，再验证当前值
- 通过标准：符合“掉电不保存”的需求口径
- 注意：若掉电前设置动作本身未成立，不进入掉电结论

#### F. FACTORY

- 测什么：恢复出厂后默认唤醒词、默认音量、默认工作模式、默认窗帘模式是否恢复
- 怎么测：先人为改离默认，再执行恢复出厂，再逐项验证默认是否回来
- 通过标准：四类默认状态恢复
- 特别规则：若功能恢复已成立，但协议或播报 ID 与需求不同，记为数值差异，不阻断后续测试

#### G. WAKEWORD

- 测什么：默认唤醒词恒有效、候选词设置外无效、设置后立即生效、未设置候选词不误唤醒、超时退出
- 怎么测：默认词验证、设置入口验证、新词验证、其他词负例验证、超时验证拆开设计
- 通过标准：新词设置后可用，默认词仍可用，其他未设置词无效
- 注意：误唤醒必须按真正唤醒成立的组合证据判定，不能只因识别文本浮现就判错

#### H. WORKMODE

- 测什么：语音模式/滴答模式切换与反馈差异
- 怎么测：进入设置工作模式，分别切两种模式，再用同一稳定控制词验证反馈变化
- 通过标准：切换成功，且反馈口径随模式变化
- 数值验证：语音模式与滴答模式的播报 ID 是否符合需求

#### I. CURTAINMODE

- 测什么：窗帘模式、纱帘/窗纱模式、布帘模式切换以及模式专有词生效范围
- 怎么测：进入设置窗帘模式，切目标模式，再验证对应专有词和非当前模式词
- 通过标准：当前模式专有词有效，其他模式专有词无效
- 注意：通用窗帘词不能直接拿来做所有模式互斥负例

#### J. CURTAIN

- 测什么：开/关/停控制命令是否正确
- 怎么测：在目标模式下分别验证打开、关闭、停止
- 通过标准：行为成立，播报和协议正确或至少功能正确
- 数值验证：协议是否为当前需求值

#### K. PHRASE

- 测什么：同义词是否等价识别
- 怎么测：同组词逐条验证，例如 `打开窗帘 / 窗帘打开`
- 通过标准：同一语义组都能落到同一类合法动作
- 注意：ASR 文本写法不同但语义等价时，功能应判通过

#### L. SELECTOR

- 测什么：定向唤醒生效、未选词不误唤醒、旧词切换后失效、默认词仍可用
- 怎么测：设置 A -> 验 A；设置 A -> 验 B 无效；切到 B -> 验 A 失效；再验默认词
- 通过标准：只有当前要求有效的词保持有效

#### M. CTRL

- 测什么：配对遥控器入口、窗口内成功/失败播报、窗口超时、窗口关闭后二次注入无效、清除遥控器
- 怎么测：入口动作与被动协议注入分开验证
- 通过标准：窗口行为和被动播报成立
- 特别规则：若入口主动协议和需求不一致，但窗口功能成立，则记数值差异，不直接阻断后续配对窗口用例

### 3.6 结果判定与归因示例

推荐按三层输出：

#### 第一层：功能结论

- `PASS`：功能成立
- `FAIL`：功能不成立
- `PARTIAL`：功能成立但数值不一致
- `DRY_RUN / NO_METHOD`：未自动执行

#### 第二层：数值差异

对功能已经成立的项，再单独列：

- 需求协议 vs 实测协议
- 需求播报 ID vs 实测播报 ID
- 需求超时 vs 实测超时
- 需求默认值 vs 实测默认值

#### 第三层：异常归因

按顺序优先判断：

1. 环境问题
2. 烧录问题
3. 观测/runner 问题
4. 用例设计问题
5. 固件功能问题
6. 需求/词表/tone 差异

### 3.7 停测规则示例

以下情况应停止正式测试并直接上报：

- 烧录模式进入失败，且完整重试后仍失败
- 烧录后 `20s` 健康检查存在自动重启或异常复位
- 最小可测性验证失败
- 日志口完全不可读
- 协议口在明确应有数据的基础链路中持续不可观测
- 设备在正常语音执行过程中发生非主动异常重启

以下情况不应阻断后续测试，应继续执行并单独记录：

- 功能成立但协议与需求不一致
- 功能成立但播报 ID 与需求不一致
- 功能成立但默认值/超时时间与需求存在差异
- 配对入口主动协议不一致，但窗口行为与被动结果正常

## 4. 给其他 skill 的复用方式

后续其他 skill 如果读取 `masr-pillar`，建议按下面顺序生成新的测试方案：

1. 先从新需求里抽项目基础信息、默认值、串口、掉电、恢复出厂、设置类、控制类、窗口类。
2. 先写“需求内容提炼”，不要直接写 case。
3. 再写“烧录门禁 + 最小可测性验证门禁”。
4. 再按功能模块写“测什么、怎么测、什么算通过、什么属于数值差异、什么属于停测”。
5. 最后才把这些方案落成用例、断言和 runner 动作。

如果新项目也属于“离线语音 + 串口协议 + 设置状态机 + 模式切换”这一类，基本都可以沿用本文档的骨架，只替换：

- 默认值
- 唤醒词/命令词
- 协议
- 播报 ID / 文案
- 模式集合
- 掉电/恢复出厂规则
- 停测门槛

## 5. 当前示例文档的作用边界

`masr-pillar` 不是最终测试报告，也不是最终用例表。

它的定位是：

- 一份“需求内容怎么抽”示例
- 一份“测试方案怎么写”示例
- 一份“烧录/预检/停测门禁怎么设计”示例
- 一份“功能与数值如何拆开判断”示例

后续如果要真正生成某一轮新项目的测试方案，应在这份文档骨架上替换成该项目自己的需求数据。
