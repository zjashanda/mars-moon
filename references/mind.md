# 需求理解与用例生成思路

## 1. 我如何理解这类需求文档

这类杜亚窗帘离线语音项目的需求文档，本质上不是一张平铺的“命令词清单”，而是一套带状态机的功能规格。阅读时我会先把信息拆成四层：

1. 基础元信息
   - 项目名、芯片、分支、固件版本。
   - 默认串口口径、日志口径、超时时间、音量档位。
2. 默认状态
   - 默认唤醒词。
   - 默认工作模式。
   - 默认窗帘模式。
   - 掉电后哪些状态保留、哪些状态恢复默认。
3. 可切换状态
   - 唤醒词候选集合。
   - 工作模式候选集合。
   - 窗帘模式候选集合。
   - 音量边界与恢复规则。
4. 动作与反馈
   - 每个词条的语义、预期播报、协议发送、协议接收、日志关键字。
   - 成功条件、失败条件、互斥关系和恢复路径。

换句话说，我不会把需求文档只当作“输入文本”，而是把它整理成“状态 + 触发 + 反馈 + 持久化”的结构化规格。

## 2. 我如何分析功能之间的逻辑

### 2.1 先找状态机

对这个项目，我会优先识别下面几条主状态轴：
- 唤醒状态：休眠、已唤醒、设置窗口中。
- 工作模式：语音模式、滴答模式。
- 窗帘模式：窗帘、纱帘/窗纱、布帘。
- 唤醒词状态：默认词或某个候选词已生效。
- 音量状态：当前档位、边界档位、掉电保存状态。
- 系统状态：刚重启、日志未恢复、可正常测试。

这些状态轴一旦交叉，很多功能就不是单点功能，而是“在某个前置状态下才成立”的功能。

### 2.2 再找依赖关系

我会把功能依赖分成几类：

- 前置依赖
  - 例如所有设置类命令前都必须先唤醒设备。
  - 进入“设置唤醒词”“设置工作模式”“设置窗帘模式”后，才允许播报目标值词。
- 模式依赖
  - 工作模式会影响唤醒后的播报口径，例如滴答模式下统一播报不再是语音模式下的 `TONE_ID_0`。
  - 窗帘模式会影响哪些名词有效、哪些名词互斥。
- 持久化依赖
  - 例如音量掉电保存必须等 `save` 相关标记出现，确认已经写入 flash 后再断电。
- 恢复依赖
  - 工厂恢复、重启、掉电后，哪些配置应该恢复默认，哪些应该保持上次值。
- 观察依赖
  - 设备一旦重启，日志等级会恢复默认，此时即便功能还在跑，也可能因为日志看不到而误判，所以必须先恢复日志等级再继续测试。

### 2.3 最后找互斥与例外

需求里最容易漏的是“默认生效”和“互斥边界”：

- 窗帘相关命令词是默认生效词，不管处于纱帘、窗纱还是布帘模式，通用窗帘词通常都仍然有效。
- 已确认的设置窗口里，普通控制词也可能继续有效。例如 `设置唤醒词 / 设置工作模式 / 设置窗帘模式` 窗口内说 `打开窗帘`，不能直接当成“无效词”。
- 这类设置态真正要验证的不是“普通控制词静默不响应”，而是“普通控制词执行后，设置窗口是否仍然可继续完成后续目标设置”。
- 真正互斥的是纱帘/窗纱模式与布帘模式对应的专有名词，而不是把所有窗帘词都互斥掉。
- 主动 `reboot` 和主动上下电是测试动作，不应算作异常重启；正常语音执行期间的被动重启才算异常。

所以我分析逻辑时，会特别把“通用词继续生效”和“专有词互斥失效”拆开建模，而不是简单做二值判断。

## 3. 我如何把测试点生成成用例

### 3.1 先按模块拆测试点

我会把需求拆成稳定模块，再为每个模块设计用例：

- `BOOT`
- `BASE`
- `VOL`
- `UART`
- `POWER`
- `FACTORY`
- `WAKEWORD`
- `WORKMODE`
- `CURTAINMODE`
- `CURTAIN`
- `PHRASE`
- `SELECTOR`
- `CTRL`

这样做的目的是让每条 case 都有清晰主题，失败后能快速归因。

### 3.2 用“一个核心断言 + 必要前置闭环”组织单条 case

单条 case 尽量只承载一个核心断言，但前置步骤必须完整。例如设置类 case 不能只写“发送设置命令”，必须展开成：

1. 唤醒设备。
2. 进入对应设置模式。
3. 播报目标词。
4. 校验成功反馈。
5. 再用后续动作验证设置真的生效。
6. 必要时恢复默认状态。

这样生成的 case 才是真正可执行、可复盘的闭环，而不是纸面用例。

额外要强调两点防假 PASS 规则：

- 正向控制命令不能只命中播报或只命中协议，默认要同时验证 `playId/tone` 和 `sendMsg/protocol`。
- 设置成功不能只看“设置成功”播报，默认还要补 `configSaved=save config success`，并通过后续行为证明配置真的落地。

### 3.3 对复杂功能做遍历，不做“只看设置成功”

最典型的是唤醒词切换。对每个候选唤醒词，设置成功后至少要做四件事：

1. 当前设置的唤醒词可以唤醒。
2. 默认唤醒词仍然可以唤醒。
3. 其他非当前候选词不能唤醒。
4. 切换完成后命令词仍然能正常工作。

也就是说，设置成功播报只说明“入口动作成功”，不代表“功能状态已正确切换”。真正的用例要继续验证功能效果。

### 3.4 对短语和同义词做分层

短语类不会简单地“一词一 case”无脑铺开，而是按业务含义分层：

- 同一模式、同一前置、同一断言的同义词，可以归并成一组短语检查。
- 但结果导出时必须把组内每个词的执行步骤展开显示，避免后续只看到“检查 N 条”而不知道具体测了什么。
- 对模式敏感的短语，必须先显式设置对应模式再测，不允许借用上一条 case 的残留状态。

### 3.5 对掉电、重启、工厂恢复做“动作后再观察”

这些 case 不是只做控制动作就结束，而是必须带恢复观察：

- `BOOT`：要区分软 `reboot` 和通过控制串口触发的硬重启；有些上电播报只会在硬重启后出现。
- `POWER`：对保存类功能，要等 `save` 标记出现或等待超时后再断电。
- `FACTORY`：恢复出厂后，不仅要看播报，还要重新验证默认状态是否真的恢复。

### 3.6 对 runner 规则做工具级兜底

用例生成只是第一层，runner 还要兜住执行期规则：

- 任何重启后都自动恢复 `loglevel 4`，确认日志恢复后再继续。
- 主动重启、主动上下电同步异常重启计数基线，避免误停测。
- 非主动重启、非主动断电一旦发生，当前用例直接判失败，不允许“日志恢复后继续算通过”。
- 连续 3 条用例因唤醒失败则停止，避免设备跑偏后继续扩大无效失败。
- 不同工作模式下使用不同的唤醒成功判定口径。
- setup / cleanup 恢复默认状态是执行保底动作，不应计入主用例的协议、tone 和 gap 摘要。

也就是说，测试逻辑分成两层：

- 生成层：决定“应该测什么、如何编排步骤”。
- 执行层：决定“现场怎么判定、何时停测、如何保现场”。

## 4. 这套方法落到当前项目时的具体理解

### 4.1 唤醒词功能

我把它理解成“配置项 + 生效验证”的组合，而不是单纯设置项。

- 默认词始终要保留验证。
- 每个候选词都要单独遍历切换。
- 每次切换后都要验证命令词链路仍可用。

### 4.2 窗帘模式功能

我把它理解成“通用词始终有效 + 专有词按模式互斥”的组合。

- 通用窗帘词默认生效。
- 纱帘/窗纱词与布帘词互斥。
- 短语验证时要先显式设置模式，再验证该模式下的词集与负例词集。

### 4.3 设置类功能

我把它理解成统一模板：

- 先唤醒。
- 再进入设置入口。
- 再播报目标值。
- 再验证状态已经切换。
- 在未超时前，如果播报的是当前模式允许的普通控制命令，也应视为可生效路径，而不是默认当无效词。
- 但普通控制命令可生效，不代表该需求自动通过；还要继续验证设置窗口没有被错误打断，并且后续目标设置仍能完成。

### 4.4 电源与日志

我把它理解成“系统状态恢复”问题，而不是单纯功能问题。

- 每次重启后日志等级会恢复默认，不先恢复日志就会影响后续判定。
- 所以 runner 里必须把“日志恢复”视作所有重启路径的固定步骤。

## 5. 当前执行时必须遵守的固定规则

这一部分不是临时经验，而是当前 `mars-moon` 后续复用时也应该继续沿用的执行规则。

### 5.1 每次正式测试前都要先做串口可用性检查

- 至少确认控制口、日志口、协议口都能被正常打开。
- 如果本轮包含烧录，还要确认烧录口存在并可用。
- 一旦中途出现串口断开、端口名变化或打开失败，不能直接继续跑 case，要先恢复串口链路。

### 5.2 每次检测到重启后，都必须重新设置日志等级

- 不管是主动 `reboot`、主动上下电，还是执行中被动重启，都要重新发送 `loglevel 4`。
- 只有在确认日志已经恢复输出后，才允许进入下一条正常测试。
- 如果没有先恢复日志，就算功能表面还在跑，也可能因为观察面丢失而误判。

### 5.2A 日志口/协议口波特率要优先跟随需求文档

- 常规无需求文档场景，可以沿用 `tools/serial_ports.json` 里的平台默认波特率。
- 常规无需求文档场景，播报声卡优先沿用 `tools/audio_devices.json` 里的平台默认 `playback_device_key`。
- 如果配置声卡失效且当前只检测到 `0~1` 个 ListenAI 播放设备，可以按 `tools/audio_devices.json` 的回退策略临时走默认声卡。
- 如果当前检测到多个 ListenAI 播放设备，禁止自动回退默认声卡；必须先让测试人员指定其他声卡或显式默认声卡，再继续测试。
- 只要本轮存在需求文档，并且已经完成 `build` 生成，就要把需求文档中的日志口波特率、协议口波特率视为当前 workspace 的准值。
- 后续 `burn`、`run`、`probe` 在没有显式命令行覆盖时，都应沿用生成产物中的这两个波特率，而不是再盲目回退到 bundle 默认值。
- 如果要临时联调不同波特率，必须通过命令行显式覆盖，并在结果中注明这是临时口径，不是需求文档口径。

### 5.3 主动重启不计入异常重启次数

- 主动 `reboot` 是测试动作，不算异常重启。
- 主动上下电、手动硬重启也是测试动作，不算异常重启。
- 只有正常语音执行过程中发生的被动重启，才计入异常重启计数。

### 5.4 不同工作模式下，唤醒后的播报口径不同

- 语音模式下，唤醒后的提示音/播报口径与默认语音模式一致。
- 滴答模式下，唤醒后的统一播报会切到滴答模式口径，不能再用语音模式的固定提示音去判定。
- 因此 runner 的唤醒成功判断必须带当前工作模式上下文，不能把所有模式都写死成同一个提示音。

### 5.5 设置类命令必须走完整闭环

设置类功能不能只做“进入设置入口”或“播报目标词”中的一步，而要按完整链路执行：

1. 先唤醒设备。
2. 再进入对应设置模式。
3. 再播报目标值。
4. 再校验设置成功反馈。
5. 最后再用实际功能验证设置已经生效。

### 5.6 上电播报相关用例要区分软重启和硬重启

- 某些上电播报只会在通过控制串口触发的硬重启或上下电后出现。
- 如果测试目标是验证“上电播报”，不能只用软件 `reboot` 代替硬重启。

### 5.7 掉电保存类用例要等待保存落盘

- 例如音量掉电保存，不是刚设置完就立刻断电。
- 应等待 `save` 相关标记出现。
- 如果一段时间内没有等到保存标记，也应按约定超时后再断电，避免因为保存尚未落盘而误判。

### 5.8 连续唤醒失败需要止损

- 如果连续 3 条用例都是因为唤醒失败而失败，应立即停测。
- 这样做是为了避免设备已经偏离正常状态后，继续放大无效失败。

### 5.9 短语与模式类验证不能借用上一条 case 的残留状态

- 每条依赖模式的 case 都要自己显式设置前置模式。
- 不能默认上一条 case 执行完后设备还保持在目标模式。

### 5.10 唤醒词切换不能只看“设置成功”

- 每个候选唤醒词切换后，至少要验证四件事：
  1. 当前设置词可以唤醒。
  2. 默认唤醒词仍可唤醒。
  3. 其他候选词不能唤醒。
  4. 切换后命令词仍然正常生效。

### 5.11 任何非主动重启都要判当前用例失败

- 只有主动 `reboot`、主动上下电、用户明确要求的断电动作，才属于预期重启。
- 其他任何用例执行中的被动重启，都要立即判当前 case `FAIL`。
- 日志恢复可以帮助继续后续测试，但不能把当前 case 改判为通过。

### 5.12 人工确认项要和自动失败项分开管理

- `CFG-006 / CFG-007 / CFG-008 / CFG-010 / CFG-014` 当前按人工确认项处理。
- 这些项应保留在需求表中，但结果口径用 `NO_METHOD / 人工确认`，不要混进自动化失败统计里。

## 6. 如何区分工具问题、固件问题和用例问题

我复盘时会优先按下面逻辑区分：

1. 如果日志里已经稳定看到 `asrKw / sendMsg / playId`，而行为与需求口径不一致，更像固件问题或需求口径问题，不像 runner 问题。
2. 如果失败来自 setup/cleanup 恢复动作污染主结果，或只验证了 tone/只验证了 protocol 就误判，则是用例/runner 问题。
3. 如果说了目标词后只有提示音、没有 `configSaved`、也没有后置行为变化，要重点判断是固件没落盘，还是“重复选择当前值是否必须保存”的口径未定。
4. 如果测试过程中出现非主动重启，则当前 case 先按固件稳定性问题处理；只有在确认证据链有误时才回看工具。

## 7. 后续如果需求变化，我会怎么改

如果后续需求改变，我会按下面顺序处理，而不是直接硬改 case：

1. 先更新这份 `mind.md` 中的方法理解，明确新增状态、互斥关系、保存规则或例外。
2. 再更新 `references/checkLogic.txt` 或补充说明中的判定口径。
3. 然后改 `tools/dooya_spec_builder.py`，让规格化结果先正确。
4. 再改 `tools/dooya_case_builder.py`，让 case 编排和步骤展开正确。
5. 最后用 `prepare -> build -> run --dry-run -> burn -> run` 再走一遍完整链路，确认变化真的落到执行结果里。

这样做的目的，是保证“需求理解 -> 规格建模 -> 用例生成 -> 执行验证”始终是一条闭环，而不是只改表面现象。
